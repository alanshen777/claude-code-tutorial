Part 0

你的电脑是怎么运作的

在碰任何 AI 工具之前，先在脑子里建立正确的"数字世界地图"。大部分小白卡住不是因为不会操作，是因为脑子里没有正确的画面。这一章帮你把画面补上。

📁 文件与文件夹

你在电脑上做的一切——写文档、存照片、装软件——最终都变成了文件，放在某个文件夹里。文件夹可以嵌套，形成一棵"树"。

类比：文件 = 一张纸，文件夹 = 抽屉。你的电脑就是一个巨大的文件柜，每个抽屉里还可以有小抽屉。
文件系统就像一棵倒过来的树：从根目录向下展开，文件夹可以包含文件夹

在电脑里，你看到的桌面、文档、下载其实都是文件夹。当你把一个 Word 文件"保存到桌面"时，实际上是把这个文件放到了"桌面"这个文件夹里。理解这一点非常重要——因为 Claude Code 就是通过这棵"树"来找你的文件的。

🖥 桌面系统 vs 命令行

你平时用电脑的方式——点图标、拖文件、双击打开——这叫图形界面（GUI）。还有另一种方式，用文字跟电脑对话，叫命令行（CLI）。

两种方式做的是同一件事——只是交互方式不同

听起来命令行很可怕？其实它更接近自然语言。想想看：

  • 图形界面：你得知道按钮在哪，菜单在哪，记住一堆操作路径
  • 命令行：你直接说"打开文档文件夹"、"列出所有文件"——说什么做什么

而 Claude Code 更进一步——你甚至不需要记命令，直接用中文告诉它你想做什么，它会翻译成电脑能听懂的指令。

🏠 本地 vs 云端

Claude Code 在本地运行，但调用云端的 AI 能力来"思考"
类比：本地 = 你家书房（私密，你说了算），云端 = 公共图书馆（随时随地能用，但东西是共享的）。

关于隐私，你需要知道的是：

  • Claude Code 运行在你的电脑上，你的文件不会被上传到云端
  • 它的"思考能力"来自云端的 AI 模型——你的问题会被发送过去，但文件本身留在本地
  • 就像你打电话问朋友一个问题：你告诉了他问题，但你桌上的文件并没有寄给他

📍 路径——文件的"门牌号"

每个文件在电脑里都有一个唯一的"地址"，叫做路径（path）。

路径就像门牌号：小区 / 楼号 / 单元 / 房间号

路径用 /（斜杠）分隔每一层。当你在 Finder（文件管理器）里点击进入"文档" → "工作" → 看到"报告.docx"时，你其实就是在沿着这个路径一层层走。

小贴士：不用记住路径怎么写。当你用 Claude Code 时，直接说"帮我看看桌面上的那个 Excel 文件"，它会自动找到对应的路径。

📦 "安装"到底在做什么

很多人对"安装"有神秘感。其实安装一个程序，本质上就是三件事：

  • 下载：把一堆文件从网上搬到你的电脑
  • 放置：把这些文件放到约定好的位置
  • 注册：告诉系统"这里有个新工具可以用了"

不同类型的"程序"其实是同一个东西，只是长相不同：

名称特点例子
应用程序有图形界面，双击打开微信、Word、Chrome
命令行工具没有界面，在终端用文字操作git、node、claude

Claude Code 是一个命令行工具——没有图标可以双击，而是在终端里输入 claude 来启动。

Part 1

准备工作

最小步骤把环境跑起来。跟着做一遍，10 分钟搞定。遇到问题不要慌——每一步我都会告诉你可能碰到什么、怎么处理。

Step 1：打开终端

终端就是那个"可以打字跟电脑对话"的窗口。Mac 上有好几种打开方式：

打开后你会看到一个窗口，里面有一行文字和一个闪烁的光标。这就是命令行——你的新工作台。

终端打开后的样子——别被它吓到，它只是在等你说话
小贴士：终端窗口的颜色、字体可能跟图不一样，这完全正常。重要的是你看到了那个闪烁的光标——它代表"我准备好了，等你输入"。

Step 2：认识三个最基础的命令

在命令行里，你只需要记住三个命令就能"走路"。就像现实生活中你只需要知道"我在哪"、"周围有什么"、"去那里"：

# 1. 我在哪？（Print Working Directory = 打印当前目录） $ pwd /Users/xiaoming # 就像问"我现在在几楼？" → "你在家目录" # 2. 这里有什么？（List = 列出内容） $ ls Desktop Documents Downloads Pictures # 就像打开抽屉看看里面有什么 # 3. 去某个地方（Change Directory = 切换目录） $ cd Documents # 就像走进"文档"这个房间 # 现在再看看自己在哪 $ pwd /Users/xiaoming/Documents # 成功！你已经在文档文件夹里了 # 想回上一层？ $ cd .. # 两个点 = 上一级。就像从房间走回走廊
命令全称作用生活类比
pwdPrint Working Directory显示当前位置"我在几楼？"
lsList列出当前文件夹内容"这个房间里有什么？"
cd 目标Change Directory进入某个文件夹"走到那个房间"
cd ..—返回上一级"退出去，回走廊"
别担心：一旦你开始用 Claude Code，这些命令你几乎不需要手动输入。因为 Claude 会帮你做这些事。但理解它们在做什么，能让你更好地跟 Claude 配合。

Step 3：安装必备工具

Claude Code 需要一个叫 Node.js 的运行环境。你可以理解为：Node.js 是"基础设施"，Claude Code 是"建在上面的应用"。

1

检查是否已有 Node.js

在终端输入 node --version。如果显示版本号（如 v18.17.0），说明已安装，跳到第 3 步。如果提示"command not found"，继续第 2 步。

2

安装 Node.js

访问 nodejs.org，下载 LTS 版本（长期支持版），双击安装包，一路"下一步"。安装完后关掉终端再重新打开。

3

安装 Claude Code

# 一行命令搞定 $ npm install -g @anthropic-ai/claude-code # 安装完验证一下 $ claude --version claude-code v1.x.x # 看到版本号就说明安装成功了！
npm 是什么？它是 Node.js 自带的"应用商店"。npm install -g 就是在说"从应用商店下载并全局安装"。-g 表示全局（global），意味着在任何文件夹都能用。

Step 4：第一次启动 Claude Code

# 先进入你想工作的文件夹 $ cd ~/Documents # 启动 Claude Code $ claude

第一次启动会要求你登录 Anthropic 账号。按提示操作即可（通常会打开浏览器让你授权）。

登录成功后，你会看到这样的界面：

启动成功！看到 "You:" 提示符就可以开始对话了

现在你可以直接用中文跟 Claude 说话了。试试输入：你好，你能做什么？

Part 2

开始使用 Claude Code

环境搭好了。现在学会跟 Claude Code 协作，理解它在做什么——这比背命令重要得多。

💬 让 Claude 帮你做第一件事

不需要学特殊语法，直接用中文说你想做什么。比如：

You: 帮我看看这个文件夹里有什么文件，按类型整理一下告诉我 Claude: 我来看看当前目录的内容... [Claude 调用了 ls 命令，读取了文件列表] 当前文件夹包含以下内容： 📁 文件夹（3个）：工作项目、个人照片、学习资料 📄 文档（5个）：简历.docx、方案.pdf、会议记录.md... 🖼 图片（2个）：截图1.png、logo.jpg

注意：Claude 不只是在"猜"——它真的去执行了命令，读取了你的文件系统。

🔍 理解 Claude 在做什么

Claude Code 和普通 AI 聊天的最大区别：它能真正操作你的电脑。它做的每一步都会告诉你。

Claude Code 的工作方式：理解你的需求 → 用工具执行 → 报告结果

每次 Claude 使用一个工具，你都能在界面上看到它做了什么。这意味着：

  • 你可以监督它的每一步操作
  • 它不会偷偷做你不知道的事
  • 如果哪一步不对，你可以立刻纠正

🔐 权限确认——Claude 在问你"可以吗"

当 Claude 要做可能有影响的操作（修改文件、执行命令），它会先问你：

Claude: 我需要修改 报告.docx 的内容，把第三段改为... Allow this edit? (y/n) # y = 同意（yes） # n = 拒绝（no），Claude 会换个方案
类比：就像你的助理拿着笔走到你面前说——"我准备把这段改了，你看行吗？"你点头它才动笔。这是一个安全机制，确保 Claude 不会做你不想做的事。

一般来说：

  • 读文件：通常自动允许（只是看看，不影响任何东西）
  • 改文件：需要你确认（会改变你的文件内容）
  • 执行命令：需要你确认（可能安装东西或执行操作）
随着信任积累：你可以在设置里把某些操作设为"自动允许"，这样 Claude 就不用每次都问你了。但刚开始建议保持手动确认，先熟悉它的行为模式。

📂 文件夹 = 你的工作空间

这是一个很重要的概念：你在哪个文件夹启动 Claude，它就主要看那个文件夹里的东西。

所以养成习惯：先 cd 到你要工作的文件夹，再启动 claude。这就像走进办公室再开始工作，而不是站在大门口工作。

Part 3 — 核心认知

人与 AI 的协作层级

这是整个教程最有价值的部分。不是教操作，是教认知。你和 AI 之间的协作深度，决定了你能从 AI 身上获得多大的价值。这里有 6 个层级——每一层的跨越都是认知跃迁，不是技术升级。

每上一层，你从 AI 获得的价值呈指数级增长
Level 0 · 工具模式

一次性问答

"帮我写一封邮件" → 拿到结果 → 结束
这是最基础的使用方式：你问一个问题，AI 给一个答案。像用搜索引擎一样——输入关键词，得到结果。大部分人使用 AI 停留在这个层级。
  • 把 AI 当搜索引擎或模板生成器
  • 一次输入，一次输出，用完即走
  • 结果质量完全取决于你的 prompt（提问方式）
  • AI 对你毫无了解——每次都是陌生人
瓶颈：你必须一次把需求说清楚，但现实中很少有人能做到。结果往往不满意，然后归结为"AI 不够聪明"。
Level 1 · 对话模式

多轮打磨

"帮我写一封邮件" → "语气再正式一点" → "加上这个背景" → 满意
你开始意识到 AI 可以被"调教"。通过多轮反馈，逐步逼近你想要的结果。这是一个重要的认知跳跃——从"一锤子买卖"到"渐进式迭代"。
  • 理解了 AI 的输出可以被迭代改进
  • 学会用反馈来引导 AI 的方向
  • 每次对话的质量明显提升
  • 但每个新话题还是从零开始
关键认知：你不需要一次说清楚。可以先给一个大方向，然后通过反馈不断调整。好的结果是"对话"出来的，不是"问"出来的。
Level 2 · 协作模式

先讨论后执行

"我想做X，但还没想清楚" → 一起讨论 → 对齐方案 → 制定计划 → 逐步执行
这是一个巨大的认知跃迁。你开始把 AI 从"执行者"升级为"思考伙伴"。在动手之前，先跟 AI 讨论——让它帮你理清思路、发现盲点、对齐方向，然后再让它去做。
  • AI 不只是执行者，还是你的思考伙伴
  • 先用 AI 帮自己理清思路（"想"），再让 AI 执行（"做"）
  • 方案质量大幅提升——因为想清楚了再做
  • 开始出现"人机配合"的感觉
关键认知：把"想"和"做"分开。AI 两个都能帮——但让它帮你想，往往比让它帮你做更有价值。
Level 3 · 委托模式

目标驱动

"我的目标是X" → AI 自己拆解计划 → 自己执行 → 自己验证 → 汇报结果
你只给目标和约束条件，不管过程。AI 自己决定怎么做、怎么拆分、先做什么后做什么。你的角色从"操作者"变成了"Review 者"——看结果，给反馈。
  • 你只定义"做什么"和"不能做什么"，不管"怎么做"
  • AI 自主决策执行路径，你不用 micromanage
  • 你的时间从"做事"释放出来，用于"判断和决策"
  • 需要学会 review —— 看得懂结果，能判断好坏
关键认知：信任但验证。让 AI 自由发挥，但保持对结果的审核。学会做"甲方"——不是什么都自己干，而是能判断"做得好不好"。
Level 4 · 系统模式

架构师思维

设计定时任务、触发规则、多 Agent 协作、自动化工作流——AI 按规则持续运转
你不再是"用 AI 做一件事"，而是在设计一套系统，让 AI 作为系统的一部分自动运转。就像写了一套流水线规则——原材料进来，成品出去，你不用每次都站在旁边。
  • 你在设计系统，不是在做具体任务
  • AI 是系统的一个组件，按规则持续运转
  • 一次设计，持续产出——效率呈指数级增长
  • 开始理解"可复用"和"自动化"的力量
关键认知：从"用 AI 做事"到"让 AI 替你运转"。你的价值不是做事，是设计让 AI 做事的系统。
Level 5 · 共生模式

你 + AI 操作系统

AI 深度嵌入你的知识管理、任务管理、决策支持系统，成为你的"外脑"
AI 不再是一个外部工具，而是你个人操作系统的核心组件。它了解你的目标、原则、偏好、历史决策，不只执行任务，还主动提供洞察和建议。你们在共同进化——你让 AI 变得更强，AI 也让你变得更强。
  • AI 了解你的完整上下文——目标、原则、偏好、工作方式
  • 不只执行任务，还主动发现问题、提供建议
  • 你的知识和经验通过 AI 被结构化、放大、持久化
  • 你和 AI 形成正反馈循环——越用越强
关键认知：AI 不是工具，是协作者——你们在共同进化。你的天花板不再是"一个人能做多少"，而是"人+AI 系统能做多少"。

每一层的跨越不是技术问题，
是认知跃迁。

市面上的教程教 Level 0-1 的操作。这份教程教你 Level 2-5 的认知。
Part 4

Claude Code 的配置体系

到了 Level 2-3，你会发现一个问题：每次新对话，Claude 都像一个"新来的同事"，啥都不知道。配置体系就是解决这个问题的——让 Claude 从"陌生人"变成"老搭档"。

四个配置层共同构建了 Claude 对你的"理解"
📋

CLAUDE.md

给 AI 的"员工手册"

放在项目文件夹根目录的一个文本文件。Claude 每次启动都会自动读取它。你可以在里面写：

  • · 你是谁、做什么工作
  • · 你的项目文件结构
  • · 你的规则和偏好
  • · 你希望 Claude 怎么配合

越写越详细 → Claude 行为越符合期望。就像一份越来越完善的员工手册——新来的同事按手册做事就能达到 80 分。

🧠

Memory

AI 的"工作笔记本"

Claude 在协作过程中自动积累的笔记——你的角色、偏好、反馈、项目进展。每次新会话开始前它都会翻看。

你也可以主动告诉 Claude"记住这个"。就像你的助理随身带着一个小本子，记着你喜欢什么、不喜欢什么、上次讨论到哪了。协作时间越长，它越"懂你"。

⚡

Command

可复用的指令模板

把重复性工作封装成一条指令。比如你每天都要做一个"日报"，可以创建一个 /daily-report command，Claude 就按固定流程帮你生成。

类比：跟助理说"按上次的格式再写一份"——它已经知道格式、步骤、注意事项。你不用每次重新解释。

🎓

Skill

能力包

比 Command 更完整——不只是步骤模板，还包含领域知识、判断规则、输出规范。让 Claude 学会一个"领域能力"。

类比：Command 是告诉助理"按步骤做"，Skill 是送助理去培训——回来后它能自己判断、灵活应对。

四者的关系

配置作用什么时候用类比
CLAUDE.md定义规则和上下文项目开始时写，持续完善公司制度
Memory跨会话记住信息自动积累 + 主动保存工作日志
Command封装重复流程第二次做同一件事时SOP 清单
Skill封装领域能力需要专业判断力时培训证书
起步建议：先写一份简单的 CLAUDE.md（哪怕只有 3 行），体验一下"Claude 知道你是谁"的感觉。然后在协作中让 Claude 帮你逐步完善。
Part 5

沉淀你的工作流

这是从"使用者"到"构建者"的跃迁——也是 Level 4 和 Level 5 的基础。不是让你学编程，而是让你学会一个习惯：做完事情后，把方法留下来。

当你发现自己第二次用类似方式解决问题时，
就应该把它沉淀下来。

第一次是尝试，第二次就是模式
沉淀的正循环：做 → 回顾 → 沉淀 → 复用 → 迭代 → 做得更好

沉淀什么到哪里？

你发现了什么沉淀为下次怎么用
解决某类问题的步骤Command/command名 一键触发
某个领域的工作方法论SkillClaude 自动具备该领域能力
跟 Claude 协作的规则CLAUDE.md每次会话自动加载
重要的背景信息MemoryClaude 跨会话记住

怎么沉淀？（具体步骤）

你不需要自己写代码或配置文件——告诉 Claude 就行。

1

做完事后问自己

"这个过程有没有可复用的部分？下次还会做类似的事吗？"

日常场景实战

用真实场景串联前面所有知识。重点展示 Claude Code 最独特的能力——自动读取文件作为上下文、多轮迭代编辑、以及作为思考伙伴帮你做方案。

📂 场景一：自动加载文件做上下文

Claude Code 最强的能力之一：它能直接读你电脑上的文件。你不用复制粘贴内容给它——直接说"看一下那个文件"，它就能读。

You: 看一下 ~/Documents/产品需求/用户反馈汇总.xlsx， 帮我按问题类型分类，找出提到最多的前 5 个痛点 [Claude 自动读取文件 → 分析内容 → 生成结构化总结] Claude: 我读了这个文件，共 238 条反馈。按类型分类后： 📊 Top 5 用户痛点： 1. 搜索结果不准确（47次，19.7%）— 用户期望精确匹配但得到模糊结果 2. 导出功能缺失（31次，13.0%）— 无法批量导出为 Excel 3. 页面加载慢（28次，11.8%）— 集中在列表页超过100条时 4. 筛选条件不够灵活（25次，10.5%）— 缺少时间范围和组合筛选 5. 移动端体验差（22次，9.2%）— 表格在手机上无法横向滚动 要不要我进一步分析每个痛点的具体用户原话？

再看一个更强的场景——跨文件加载上下文：

You: 我要给客户做一份项目汇报。 先看一下这三个文件了解背景： - 项目计划书.docx - 上周的会议纪要.md - 本月的进度跟踪表.xlsx 然后帮我写一份汇报大纲 [Claude 读取 3 个文件 → 理解项目全貌 → 生成汇报] Claude: 我读完了三个文件，整理一下关键信息： - 项目目标：Q2 上线智能推荐模块 - 上周会议确定：推荐算法从协同过滤改为向量检索 - 进度：整体 68%，算法模块滞后 1 周 建议汇报大纲： 1. 项目概况与里程碑回顾 2. 本月关键进展（3 项） 3. 技术方案调整说明（协同过滤 → 向量检索，为什么） 4. 风险项：算法模块滞后的应对方案 5. 下一步计划与资源需求
核心区别：普通 AI 聊天（ChatGPT / Claude.ai 网页版）需要你复制粘贴内容进去。Claude Code 直接读文件——它拿到的是完整上下文，不是你截取的片段。这意味着它的理解更准确，你也更省事。

另一个常见场景——修改已有的长文档：

🧠 场景三：多轮做方案 / 设计

这是 Level 2（协作模式）的典型场景。你不需要一开始就想清楚——先跟 Claude 讨论，逐步收敛到方案。

── 第 1 轮：抛出模糊想法 ── You: 我想给团队做一个内部知识库。现在大家的经验都在各自脑子里， 新人入职要问一圈才知道流程。但我还没想清楚具体怎么做。 Claude: 好问题。在开始之前，我想先了解几个关键信息： 1. 团队多大？（5人 vs 50人方案完全不同） 2. 现在知识散落在哪些地方？（飞书文档/Notion/微信群/口口相传） 3. 最痛的场景是什么？（新人入职？还是日常查找？） 4. 你希望维护成本高还是低？（有人专门维护 vs 自然积累） ── 第 2 轮：提供背景 ── You: 12 人的产品技术团队。知识在飞书文档里但很散， 没有结构。最痛的是新人——上个月来了 2 个人， 我花了整整 3 天给他们讲各种流程。维护成本要低。 Claude: 明白了。根据你的情况，我建议的方案： 方案：飞书文档 + 结构化索引（而不是另起一个工具） 理由： - 团队已经在用飞书，不增加工具切换成本 - 12 人规模不需要专业知识库工具（过重） - 核心问题不是"没有文档"，是"文档没有结构" 具体做法： 1. 建一个"团队知识库"根文档，作为入口 2. 按 4 个分类组织：流程类 / 规范类 / 工具类 / FAQ 3. 每个分类下用飞书的"目录"功能链接到已有文档 4. 新人入职 checklist 单独一页，链接到相关流程文档 要不要我帮你出一个具体的目录结构？ ── 第 3 轮：细化执行方案 ── You: 好，出一下目录结构。另外"流程类"里最重要的是 发版流程和 bug 处理流程，这两个能帮我先写出来吗？ Claude: 我先出目录结构，然后写两个流程文档。 [Claude 创建 知识库目录.md → 创建 发版流程.md → 创建 Bug处理流程.md] 已创建 3 个文件 ✅ 你看看内容是否准确，我可以继续调整。

再看一个产品设计的多轮协作：

── 第 1 轮：从问题出发 ── You: 我们的搜索功能用户满意度很低。先看一下 ~/Documents/产品/搜索功能-用户反馈.md 和 ~/Documents/产品/搜索功能-埋点数据.csv， 帮我分析问题出在哪 [Claude 读取两个文件] Claude: 分析完成。核心问题不是搜索"不准"，而是： 1. 用户期望精确匹配，但系统做的是模糊搜索 → 埋点显示：72% 的搜索查询是完整标题或 DOI 2. 无结果时没有引导 → "未找到结果"页面的跳出率 89% 3. 筛选条件使用率极低（3.2%），因为默认折叠了 ── 第 2 轮：一起想方案 ── You: 第 1 点很关键。那你觉得应该怎么改？ 既要保留模糊搜索的能力，又要让精确匹配的用户满意 Claude: 两种方案各有利弊： 方案 A：自动识别查询类型 - 输入像 DOI → 精确匹配 - 输入像自然语言 → 模糊搜索 - 优点：用户无感知，体验最好 - 缺点：识别可能出错 方案 B：双模式切换 - 默认精确搜索，旁边一个"智能搜索"开关 - 优点：用户有掌控感 - 缺点：多一步操作，认知负担 我倾向方案 A，因为你的埋点数据显示 72% 是精确查询， 识别规则可以做得很简单（正则匹配 DOI 格式 + 标题长度阈值）。 ── 第 3 轮：落地为文档 ── You: 同意方案 A。帮我写一份搜索优化方案文档， 保存到产品需求文档夹里 Claude: 已创建 ~/Documents/产品/搜索优化方案-v1.md ✅
核心模式：模糊想法 → 讨论澄清 → 对齐方案 → 落地为文件。你不用从一开始就想清楚——Claude 帮你从模糊走向清晰，而且全程可以调用你电脑上的真实数据做依据。
